Load Balancing

Verified: May 2026 · Sources: 7 references
A traffic cop between clients and servers — distributing requests so no single machine drowns while others idle.

2 Why It Matters

Every server has a hard ceiling — a point where CPU, RAM, or network bandwidth saturates. Load balancing shatters that ceiling by spreading work across a fleet, giving you three compounding guarantees simultaneously.

  • High Availability — when a server fails, the LB detects it via health checks and stops routing to it. Users see no downtime.
  • Horizontal Scalability — add more servers to the pool without touching DNS, client code, or any public address.
  • Performance — requests land on the least-loaded node, cutting tail latency spikes that would hit a single overloaded server.
Real-World Scale Google's frontend load balancers handle millions of requests per second globally. Netflix uses Zuul + Ribbon for service-to-service LB across thousands of microservice instances. Cloudflare uses BGP Anycast so a single IP routes to the nearest of 300+ PoPs.

3 Core Concepts

Load Balancer

Network device or software that sits between clients and servers, accepting all incoming connections and forwarding them to a backend pool.

Analogy: Airport check-in kiosk routing passengers to available agents.

Backend Pool

The group of servers (nodes) that actually process requests. The LB selects from this pool per-request based on its algorithm.

Analogy: The check-in agents — add more agents to handle a flight surge.

Health Check

Periodic probe (HTTP GET /health, TCP connect) the LB sends to each server. Consecutive failures mark a node "down" and remove it from rotation.

Analogy: Supervisor walking by to confirm each agent is at their desk.

Sticky Sessions

Routing the same client to the same backend across multiple requests — via cookie or IP hash. Required only when servers hold local session state.

Analogy: Always returning to the same agent who already knows your booking.

L4 vs L7 Balancing

Layer 4 routes by IP+port (fast, minimal work). Layer 7 reads HTTP headers, URL, cookies (smarter routing, TLS termination, higher CPU cost).

Analogy: L4 = sort mail by zip code. L7 = read the letter and route by contents.

Virtual IP (VIP)

Single stable IP exposed to clients. All backends can be swapped, scaled, or re-addressed without clients ever noticing.

Analogy: Company's main phone number — whoever picks up changes, the number doesn't.

4 How It Works

Every request from a client follows this deterministic path through a load-balanced system:

1

DNS Resolution → VIP

Client resolves api.example.com and receives the VIP — the load balancer's IP, not any backend's address. All N backends are hidden behind this single address.

2

Connection Established at LB

TCP handshake (or TLS handshake for HTTPS) terminates at the LB. For L7 LBs, TLS decryption happens here — backends receive plain HTTP, centralizing certificate management.

3

Algorithm Selects a Backend

In microseconds, the LB consults its routing algorithm and the current health + load state of all servers to pick a target. See the simulator below to watch this happen live.

4

Request Proxied to Backend

LB forwards the request to the chosen server. For L7 it adds X-Forwarded-For: <client-ip> so the backend can still access the real client IP for logging and rate-limiting.

5

Backend Responds

Server processes the request and sends a response back through the LB (or directly to client in Direct Server Return mode, bypassing the LB on the return path for large responses).

6

Continuous Health Check Loop

In parallel, the LB probes each backend (e.g. GET /health every 5s). Fail 3 in a row → server removed from pool. Pass 2 consecutive → server re-added. Users are never routed to a broken server.

5 Examples

Nginx — Round Robin (default)

# nginx.conf — upstream block defines the backend pool
upstream api_backend {
    server 10.0.0.1:8080;   # weight defaults to 1
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass         http://api_backend;
        proxy_set_header   X-Forwarded-For  $proxy_add_x_forwarded_for;
        proxy_set_header   Host             $host;
    }
}

Nginx — Weighted Least Connections

# Weighted + least_conn: high-capacity server gets 3x share,
# but still routes to least-busy node within its weight window
upstream api_backend {
    least_conn;
    server 10.0.0.1:8080 weight=3;   # 3x capacity
    server 10.0.0.2:8080 weight=1;
    server 10.0.0.3:8080 backup;     # only used when others are down
}

upstream api_backend {
    ip_hash;                          # sticky sessions via IP hash
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
}

HAProxy — Health Checks + Stats

# haproxy.cfg
frontend http_front
    bind *:80
    default_backend http_back

backend http_back
    balance     roundrobin
    option      httpchk GET /health HTTP/1.1\r\nHost:\ localhost
    http-check  expect status 200
    server web1 10.0.0.1:8080 check inter 5s rise 2 fall 3
    server web2 10.0.0.2:8080 check inter 5s rise 2 fall 3
    server web3 10.0.0.3:8080 check inter 5s rise 2 fall 3

# Stats dashboard at :8404/stats
listen stats
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 5s

AWS ALB — Path-Based Routing (Terraform)

# Two listener rules route /api/* and /static/* to different target groups
resource "aws_lb_listener_rule" "api" {
  listener_arn = aws_lb_listener.https.arn
  priority     = 100

  condition {
    path_pattern { values = ["/api/*"] }
  }
  action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.api_servers.arn
  }
}

resource "aws_lb_listener_rule" "static" {
  listener_arn = aws_lb_listener.https.arn
  priority     = 200

  condition {
    path_pattern { values = ["/static/*"] }
  }
  action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.cdn_servers.arn
  }
}

Consistent Hashing Ring (Python)

# consistent_hash.py
# Used for cache/shard LB: adding a server only remaps ~1/N keys
import hashlib

class HashRing:
    def __init__(self, nodes, replicas=150):
        self.ring = {}
        for node in nodes:
            for i in range(replicas):
                h = int(hashlib.md5(f"{node}:{i}".encode()).hexdigest(), 16)
                self.ring[h] = node
        self.keys = sorted(self.ring)

    def get(self, key):
        h = int(hashlib.md5(key.encode()).hexdigest(), 16)
        for k in self.keys:
            if h <= k:
                return self.ring[k]
        return self.ring[self.keys[0]]  # wrap around

ring = HashRing(["cache-1", "cache-2", "cache-3"])
print(ring.get("user-session-42"))  # deterministic, stable across requests

6 Common Mistakes

Mistake: Stateful Backends + Sticky Sessions Without Fallback If the pinned server dies, all its sessions are gone — every affected user gets logged out or loses cart state. Fix: store session in Redis/DB so any server can handle any request. Sticky sessions become optional, not load-bearing.
Mistake: Health Checks Too Infrequent or Too Shallow Checking every 30s means 30s of errors before a dead server is removed. And GET / returning 200 doesn't mean the database connection is alive. Fix: 5–10s intervals, check=2-fail=3 thresholds, and a /health endpoint that pings the DB.
Mistake: Single Load Balancer (SPOF) The LB itself is now the single point of failure. Fix: deploy an Active-Passive pair with keepalived/VRRP. The VIP floats to the standby node in under a second when the primary fails.
Mistake: Round Robin for WebSockets / Long-Polling Round Robin distributes new connections evenly but long-lived connections pile up on early servers. Server 1 opened at t=0 still has 1000 connections when Server 3 joins. Fix: use Least Connections for any workload where connection durations vary.
Mistake: Trusting X-Forwarded-For Without Allowlisting A client can forge X-Forwarded-For: 1.2.3.4 to spoof any IP for rate-limit bypass or geo-bypass. Fix: only trust the X-Forwarded-For header when the request comes from known LB IPs. Everything else: read from REMOTE_ADDR.
Mistake: Weighted RR With Stale Weights You set weights at deploy time based on instance type. Six months later half the fleet was upgraded but weights weren't updated. Fix: automate weight derivation from vCPU count, or switch to Least Connections which self-balances dynamically.

7 Cheat Sheet

Load Balancing — Quick Reference
Round Robin 1→2→3→1→… Fixed rotation, ignores load. Best: uniform stateless servers.
Weighted Round Robin Server A w=3 gets 3× requests vs w=1. Best: mixed instance types.
Least Connections New request → server with fewest open connections. Best: variable-length requests.
Least Response Time Routes to lowest (active_conns × avg_latency). Best: latency-critical APIs.
IP Hash hash(client_ip) % N. Same IP always hits same server. Best: sticky sessions without cookies.
Consistent Hash Ring topology. Add/remove server remaps only ~1/N keys. Best: distributed caches, sharded DBs.
L4 (TCP) LB Routes by IP+port. Fast, no HTTP parsing. Tools: AWS NLB, HAProxy (tcp mode), LVS.
L7 (HTTP) LB Reads URL, headers, cookies. TLS termination. Tools: AWS ALB, Nginx, HAProxy (http mode), Envoy.
Health check config Interval 5–10s · fail threshold 2–3 · rise threshold 2 · endpoint: GET /health checks DB ping.
TLS Termination LB decrypts HTTPS → plain HTTP to backends. One cert at LB, not on every server.
X-Forwarded-For LB adds original client IP. Read in app for logging/rate-limiting. Trust only from known LB IPs.
Nginx algorithms round_robin (default) · least_conn · ip_hash · hash $uri
HAProxy balance roundrobin · leastconn · source · uri · hdr(name) · random
AWS ALB L7. Path/host/header routing. Auto-scaling target groups. Free TLS via ACM. HTTP/HTTPS/gRPC.
AWS NLB L4. Static IPs, ultra-low latency, TCP/UDP/TLS. Use when L7 overhead matters or need fixed IP.
VRRP / keepalived HA protocol for LB pairs. Floating VIP moves to standby in <1s if primary fails. Eliminates SPOF.
DSR (Direct Server Return) Backend responds directly to client, bypassing LB on return path. Halves LB bandwidth for large responses.

8 What to Learn Next

🔀

Service Mesh (Envoy/Istio)

LB moves into the sidecar — per-request telemetry, circuit breaking, mTLS between microservices, no code changes.

🌐

DNS Load Balancing

Multiple A-records + low TTL. Geographic routing via Route 53 Latency/Geolocation policies. Coarser but infinitely scalable.

📡

CDN & BGP Anycast

Same IP announced from 300+ PoPs. Client's router picks the nearest one. Load balancing at the routing protocol layer.

⚡

Rate Limiting & Circuit Breakers

Protect backends from overload. Token bucket + sliding window rate limits. Hystrix/Resilience4j circuit breakers stop cascade failures.

🗄️

Database Load Balancing

Read replicas behind PgBouncer/ProxySQL. Write goes to primary, reads to N replicas. Scales read-heavy workloads linearly.

☸️

Kubernetes Ingress

K8s Services (ClusterIP, NodePort, LoadBalancer) + Ingress controllers (nginx, Traefik, Envoy Gateway) do LB natively in cloud environments.

Use wrk -t4 -c100 -d30s http://localhost/ to load test and watch the Nginx access log to verify distribution.
